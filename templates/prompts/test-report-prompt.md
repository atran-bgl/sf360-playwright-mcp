# SF360 Test Report Generator

Generate comprehensive test execution report.

---

## Hard Constraints

1. **Read all inputs** - Plan, test file, evaluation result
2. **Appropriate tone** - Positive for PASS, helpful for FAIL
3. **Actionable next steps** - Always provide clear actions
4. **Evidence for failures** - Include screenshots, logs

---

## Inputs

- `testFile`: {{TEST_FILE}}
- `planFile`: {{PLAN_FILE}}
- `evaluationResult`: {{EVALUATION_RESULT}}

---

## Required Flow

### 1. Read Inputs

Read plan, test code, and evaluation result.

### 2. Determine Tone

**If PASS:**
- Positive, celebratory ✅
- Highlight efficiency

**If FAIL:**
- Neutral, helpful ⚠️/🔧
- Categorize error
- Provide recovery steps

### 3. Generate Report Structure

```markdown
# Test Report: {testName}

**Status**: {Emoji + Status}
**Generated**: {ISO-8601}
**Duration**: {duration}

---

## Test Overview

**Page**: {pageName} ({pageKey})
**Requires Fund**: {Yes/No}
**Requires Member**: {Yes/No}
**Original Request**: {user spec}

**Test Steps**:
1. Setup: Authenticate and create fund/member
2. Actions...
3. Assertions...

---

## Execution Result

{Result details - see templates below}

---

## Test Details

**Test File**: `{testFile}`

**Code Summary**:
- Lines of code: {count}
- Uses setupTest() for authentication
- Creates unique test data with timestamps

**Key Actions**:
1. {action 1}
2. {action 2}

---

## Next Steps

{Instructions - see templates below}

---

## Artifacts

- **Plan**: `{planFile}`
- **Test**: `{testFile}`
- **Report**: `tests/reports/{testName}-report.md`
- **Screenshots**: `{if any}` *(if failure)*

---

*Report generated by SF360 Playwright MCP*
```

---

## Execution Result Templates

### If PASS (No Fixes)

```markdown
## Execution Result

✅ **Test PASSED on first attempt**

No fixes required - test executed successfully.
```

### If PASS (With Fixes)

```markdown
## Execution Result

✅ **Test PASSED after {N} automatic fixes**

**Duration**: {duration}
**Attempts**: {attempts}

### Fixes Applied

1. **{Fix 1 Title}**
   - Issue: {issue}
   - Fix: {fix}

2. **{Fix 2 Title}**
   - Issue: {issue}
   - Fix: {fix}
```

### If FAIL (Application Bug)

```markdown
## Execution Result

⚠️ **Application Bug Detected**

**Message**: {error message}

**Evidence**:
- {evidence item 1}
- {evidence item 2}
- Screenshot: {path}

### Recommended Actions

1. Verify {specific check}
2. Check if {specific check}
3. Report bug to development team with evidence above
```

### If FAIL (Setup Issue)

```markdown
## Execution Result

🔧 **Setup Issue**

**Error**: {error message}

### How to Fix

1. {step 1}
2. {step 2}
3. Re-run test: `npx playwright test {testName}`
```

---

## Next Steps Templates

### If PASS

```markdown
## Next Steps

### Run Test Again

```bash
npx playwright test {testName}
```

### Run with UI Mode

```bash
npx playwright test {testName} --ui
```

### Debug Test

```bash
npx playwright test {testName} --debug
```
```

### If FAIL (Test Code Issue)

```markdown
## Next Steps

1. Review the failing selector in test file
2. Run with debug mode: `npx playwright test {testName} --debug`
3. Use Playwright Inspector to find correct selector
4. Update test code and re-run
```

### If FAIL (Application Bug)

```markdown
## Next Steps

1. Report bug to development team
2. Include screenshots and logs from this report
3. Wait for fix before re-running test
4. Consider adding this as a known issue
```

### If FAIL (Setup Issue)

```markdown
## Next Steps

1. Follow setup instructions above
2. Run: `node helpers/verify-setup.js`
3. Re-run test after fixing configuration
```

---

## Output (Strict)

### Write Report File

Create directory: `tests/reports/` (if not exists)

Write to: `tests/reports/{testName}-report.md`

### Return Result

**Success:**
```json
{
  "success": true,
  "reportFile": "tests/reports/{testName}-report.md",
  "summary": "✅ Test PASSED in 4.5s (2 auto-fixes applied)"
}
```

---

## Summary Formats

**PASS (no fixes):**
```
✅ Test PASSED in {duration}s
```

**PASS (with fixes):**
```
✅ Test PASSED in {duration}s ({N} auto-fixes applied)
```

**FAIL (test code):**
```
❌ Test FAILED (test code issue) - Could not fix after {N} attempts
```

**FAIL (app bug):**
```
⚠️ Test FAILED (application bug detected) - See report for evidence
```

**FAIL (setup):**
```
🔧 Test FAILED (setup issue) - {error message}
```

---

## Quality Rules

1. **Concise** - Reports should be scannable
2. **Actionable** - Always provide clear next steps
3. **Evidence** - Include screenshots/logs for failures
4. **Clear categorization** - Make error category obvious
5. **Positive for passes** - Celebrate successes
6. **Helpful for failures** - Guide user to resolution
